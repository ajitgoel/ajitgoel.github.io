Empowering Product Owners: Building a Custom Configuration Editor

April 16, 2026 • 12 min read • Architecture Deep Dive

One of the most common bottlenecks in enterprise application development is the reliance on engineers for minor business logic changes. At Safeway, we faced this exact challenge: product owners needed the ability to change page zones, swap components, and apply complex business rules without waiting for a development sprint cycle. Our goal was to build a tool that empowered them to manage these configurations independently, through a clean, visual interface.

The Challenge

The technical hurdle was two-fold: providing a user-friendly frontend for complex rule-building while ensuring the output could be translated into a format our Java backend could process—specifically, Spring Expression Language (SpEL). We needed to bridge the gap between a dynamic React frontend and a robust, typed backend configuration.

Evaluating the Ecosystem

We evaluated three primary paths to deliver this configuration editor:

Solution 1

react-awesome-query-builder

A feature-complete library with a powerful rule-building engine. While it offered significant out-of-the-box functionality, its large dependency footprint and rigid UI structure made it less ideal for our specific needs.

Solution 2

react-querybuilder

A lightweight, highly customizable alternative. It provided the perfect balance of core functionality and extensibility, allowing us to tailor the UI exactly to our product owners' requirements with minimal dependencies.

Solution 3

Custom-Built Editor

We considered building a tool from scratch. However, the estimated development and maintenance effort far outweighed the benefits of leveraging a mature open-source foundation.

The Choice: react-querybuilder

We selected react-querybuilder for its modularity and ease of customization. It allowed us to focus on the most critical part of the project: the transformation logic that converts JavaScript rule structures into Java-compatible formats.

Technical Implementation

Our approach involved a tiered transformation pipeline:

  1. Frontend Capture: Users interact with the `react-querybuilder` UI to define components and rules.
  2. JS Format: The rules are exported in a standardized JavaScript/JSON format.
  3. Backend Marshalling: A custom service in our Java backend consumes the JSON and transforms it into SpEL (Spring Expression Language) strings.
  4. Validation: The generated SpEL is validated against the application context before being persisted to the database.

This "headless" configuration approach allows the UI to remain incredibly flexible while ensuring the backend receives perfectly formatted rules every time.

Outcome

The implementation is now complete and has been successfully validated through end-to-end testing. By choosing a balanced open-source solution and building a robust bridge to our Java backend, we’ve shifted the power of page configuration from developers to product owners, significantly reducing turnaround time for business logic updates.
